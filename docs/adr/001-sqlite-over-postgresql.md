# ADR-001: 选择 SQLite 而非 PostgreSQL

**状态**：已接受  
**日期**：2026-03-06  
**决策者**：Keeper 开发团队  
**相关文档**：[schema.md](../schema.md)

---

## 背景与问题

Keeper 密码管理器需要一个数据库来存储：
- 书签数据（Bookmark）
- 标签（Tag）
- 关联关系（Relation）
- 用户认证信息（Authentication）

需要在以下数据库方案中做出选择：
- **SQLite**：嵌入式数据库，单文件存储
- **PostgreSQL**：功能完整的关系型数据库
- **MySQL/MariaDB**：传统关系型数据库

---

## 决策

**选择 SQLite 作为 Keeper 的数据库引擎**

---

## 理由

### 1. 使用场景分析

**Keeper 的特性**：
- 单用户场景（本地运行）
- 无并发写入需求
- 数据量小（个人密码通常 < 1000 条）
- 本地优先（无云同步）

**SQLite 的优势**：
- 零配置：无需安装数据库服务器
- 单文件：整个数据库就是一个 `.db` 文件
- 跨平台：SQLite 几乎支持所有操作系统
- 轻量级：无后台进程，内存占用低

### 2. 性能对比

| 操作 | SQLite | PostgreSQL |
|------|--------|-----------|
| 读取 1000 条书签 | ~10ms | ~15ms（需网络往返） |
| 插入单条记录 | ~1ms | ~5ms（需网络往返） |
| 全文搜索 | FTS5 支持 | 更强大但对本场景过度 |

**结论**：SQLite 性能完全满足需求，PostgreSQL 的额外性能对单用户场景无价值。

### 3. 备份与恢复

**SQLite**：
```bash
# 备份
cp keeper.db keeper_backup_20260306.db

# 恢复
cp keeper_backup_20260306.db keeper.db
```

**PostgreSQL**：
```bash
# 备份
pg_dump keeper > keeper_backup.sql

# 恢复
psql keeper < keeper_backup.sql
```

**结论**：SQLite 备份更简单，用户可以直接复制文件。

### 4. 部署复杂度

**SQLite**：
- 无需额外安装
- 无需配置
- 无需管理数据库进程

**PostgreSQL**：
- 需安装 PostgreSQL 服务器
- 需配置用户权限
- 需管理数据库进程（启动/停止）
- 需处理端口冲突

**结论**：SQLite 零配置，降低用户使用门槛。

### 5. 数据完整性

**SQLite**：
- 支持事务（ACID）
- 支持外键约束（需启用）
- 支持触发器
- 单文件避免数据分散

**PostgreSQL**：
- 更强大的约束系统
- 更复杂的事务隔离级别
- 但对单用户场景过度

**结论**：SQLite 的 ACID 保证足够。

## 权衡

### SQLite 的局限性

| 局限 | 影响 | 缓解措施 |
|------|------|---------|
| 并发写入性能差 | ❌ 无影响（单用户） | 无需缓解 |
| 不支持网络访问 | ❌ 无影响（本地运行） | 无需缓解 |
| 不支持用户权限 | ❌ 无影响（单用户） | 操作系统文件权限 |
| 最大数据库大小 281TB | ❌ 无影响（预计 < 10MB） | 无需缓解 |
| 无内置复制/高可用 | ❌ 无影响（本地场景） | 用户手动备份 |

### PostgreSQL 的优势（对本项目无价值）

| PostgreSQL 优势 | 是否需要 |
|----------------|---------|
| 复杂查询优化 | ❌ 查询简单（主键/索引） |
| 高并发处理 | ❌ 单用户无并发 |
| 分布式部署 | ❌ 本地运行 |
| 用户权限管理 | ❌ 单用户 |
| 丰富的扩展生态 | ❌ 无需扩展 |

## 决策后果

### 正面影响

1. **部署简单**：
   - 用户只需运行 FastAPI 应用
   - 无需安装额外数据库软件
   - 降低技术门槛

2. **备份方便**：
   - 用户可直接复制 `.db` 文件
   - 支持加密备份（加密整个文件）
   - 便于迁移到其他设备

3. **性能优秀**：
   - 无网络延迟（嵌入式）
   - 读取速度快
   - 内存占用低

4. **开发效率高**：
   - Python 内置 `sqlite3` 模块
   - 使用 `aiosqlite` 实现异步操作
   - 无需配置连接池

### 负面影响（及缓解措施）

1. **无法扩展到多用户**：
   - **影响**：如果未来需要云同步/多用户，需迁移数据库
   - **缓解**：当前明确为单用户场景，未来可导出 JSON 后导入新数据库

2. **并发写入限制**：
   - **影响**：多个进程同时写入会锁定
   - **缓解**：Keeper 设计为单进程应用

3. **JSON 字段操作不便**：
   - **影响**：`urls`、`accounts` 等 JSON 字段需手动解析
   - **缓解**：使用 Python 的 `json` 模块处理

## 替代方案

### 方案 A：PostgreSQL

**优势**：
- 更强大的查询能力
- 更好的并发支持
- JSON/JSONB 原生支持

**劣势**：
- 需要安装配置
- 网络延迟
- 资源占用高
- 对单用户场景过度设计

**拒绝理由**：部署复杂度高，收益低。

---

### 方案 B：MySQL/MariaDB

**优势**：
- 流行度高
- 社区支持好
- JSON 字段支持

**劣势**：
- 同样需要独立服务器
- 配置复杂
- 对本场景无明显优势

**拒绝理由**：与 PostgreSQL 类似的问题。

---

### 方案 C：嵌入式 KV 数据库（LevelDB, RocksDB）

**优势**：
- 性能极高
- 嵌入式（无需服务器）

**劣势**：
- 无 SQL 查询
- 需要手动实现索引
- 无事务保证（部分支持）
- 无关系型约束

**拒绝理由**：开发复杂度高，无 SQL 查询能力。

## 验证与监控

### 验证指标

- [ ] **性能**：搜索 1000 条书签 < 50ms
- [ ] **稳定性**：数据库文件无损坏（运行 `PRAGMA integrity_check`）
- [ ] **备份**：文件复制成功率 100%

### 监控计划

- 定期运行 `PRAGMA integrity_check` 验证数据完整性
- 记录查询耗时（开发阶段）
- 用户反馈备份恢复体验
